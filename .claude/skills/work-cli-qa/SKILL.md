# Work CLI QA

Debug and verify the source `work` CLI in this repository.

## When To Use

Use this skill when the task touches the repo source behind:

- `../../../cli/cmd/work`
- `../../../cli/internal/work`
- `../../../cli/internal/workcli`
- `.work/` storage semantics generated by this repo
- typed work item behavior under `.work/types/` and `.work/spaces/`

Do not use this skill to operate an application repo's `.work/` tracker. Use
`work-cli` for normal work ledger operations.

## First Actions

1. Read `references/checks.md` for the current QA contract.
2. Run the repo-local QA harness from the `work-cli` root:

```bash
task qa
```

3. For machine-readable output:

```bash
task qa -- --json
```

4. To keep the temporary QA store after a successful run:

```bash
task qa -- --keep
```

5. To write a release-smoke report:

```bash
task qa -- --report /tmp/work-cli-qa.md --json
```

The harness always preserves the temporary store on failure and prints its
path so the failing `.work/` state can be inspected.

## What The Harness Owns

The skill-local CLI at `cli/main.go` builds `cli/cmd/work` from the current
checkout, then runs black-box commands against an isolated temporary store.
It installs the tracked `references/work-types/work-cli-qa/` template into the
temporary QA ledger before recording smoke work items.

It checks:

- source build health
- JSON output shape for `version`, `init`, `new`, `inbox`, `triage accept`,
  `view`, and `show`
- base layout and the built-in `research` work type preset
- flat work item files under `.work/items/*.yaml`
- inbox acceptance semantics
- status-based views
- missing work type failures do not allocate work IDs
- typed work item workspace scaffolding under `.work/spaces/<work-id>/`
- typed work through both direct create and inbox triage
- one typed QA ledger work item per smoke scenario
- Markdown release-smoke reports via `--report`
- absence of legacy top-level layouts such as `.work/views.yaml` and
  `.work/events/`

## Self-Evolution Loop

This skill is self-evolvable. When a new `work` CLI failure mode, chaotic
real-world scenario, or release concern appears, enrich the QA playbooks first,
then execute and evaluate the harness before claiming confidence.

For every new scenario:

1. Add the concise playbook case under `references/`.
2. Encode the case in `cli/main.go` when it can be made deterministic.
3. Run `task qa -- --report /tmp/work-cli-qa.md --json`.
4. Inspect the report and QA ledger work items.
5. Refine the implementation, harness, or playbook until the scenario passes.
6. Update `references/checks.md` and the release playbook when the contract
   changes.

Do not mark release readiness until the newly enriched scenario is executed and
evaluated as passing.

## Boundaries

- Keep this as a black-box QA harness. Unit tests still belong under `cli/`.
- Do not point this harness at a user's real `.work/` store.
- Do not add compatibility checks for retired tracker layouts unless a
  migration document explicitly requires them.
- If the work CLI command surface changes, update `references/checks.md`, the
  skill-local CLI, and this skill's examples in the same change.

## Release Playbook

Read `references/release-smoke-playbook.md` before using this as a release
gate.

Read `references/chaos-smoke-playbooks.md` when designing new smoke scenarios
or debugging real-world failures.
