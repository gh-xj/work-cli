# Core Beliefs

This is the philosophy of `work`:

1. Record what was done; do not perform it. The agent calls; the tool answers.

2. Trust the file on disk. It is the contract — older than any flag, more public than any CLI option.

3. Forget between calls. The binary opens, reads, writes, exits; the disk is the truth.

4. Make the work live in the repo. The disk you own is the only authority — no server, no account, no shared database.

5. Treat names as opaque. Identity is a string; a schema that names a runtime borrows its lifetime.

6. Keep facts distinct. Lifecycle, claim, attempt, and collaboration are four facts and four records; a field that means two things means neither.

7. Make views questions, not things. A saved view is a saved filter, not a saved answer.

8. Make capture cheap and commitment deliberate. The inbox is for thinking; the work item is what survived thinking.

9. Build the smallest primitive first. Sugar follows mechanism, not the other way around.

10. Keep every action reversible. A path with no return has a fault at its origin.

`work` will not become a service, an orchestrator, a workflow engine, a team SaaS, or a policy enforcer. Each would invert one of the rules above.

These rules are deliberately few. When a change cannot be squared with one of them, do not amend the rule — name the conflict, and decide which gives.
